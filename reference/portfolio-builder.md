# reference/portfolio-builder.md — 마스터 경험 뱅크 & 포트폴리오 빌더 (P0–P8)

> 이 스킬의 **기초 job**: 사용자의 모든 경험을 한 번 정리해 **마스터 경험 뱅크**를 만들고,
> 거기서 JD별 맞춤 문서(이력서·자소서·포트폴리오)를 파생한다. SKILL.md §6 ②(문서작성)의
> 상세 절차이자, **①–⑦ 전 태스크의 공통 근거 소스**(SKILL §6.0 계약). 취업 컨설턴트 방법론은 `methodology.md`,
> 채점은 `evaluation.md`, 문체는 `writing-voice.md` 참조.

---

## 0. 전역 원칙 · 런타임 다이얼

**두 축을 항상 먼저 고정한다** — ① 대상(마스터본 / 특정 JD 맞춤) ② 출력 모드(작업용 / 제출용).

**런타임 다이얼(사용자가 세션 중 조절):**

| 다이얼 | 값 | 기본 | 영향 |
|---|---|---|---|
| 출력 모드 | 작업용 · 제출용 | 작업용 | 작업용=평가·근거·증빙·자기평가 메모 노출 / 제출용=본문만, 메모·증빙 자동 숨김 |
| 언어 | 국문 · 국/영 병기 | 국문 | 병기 시 헤드라인·섹션·직무명에 영문 보조 |
| 평가 강도 | 코칭 · 균형 · 압박 | 균형 | 톤만 변경, 채점 보수성은 고정(evaluation.md §7) |
| 꼬리질문 경중 | 상 · 중 · 하 | 중 | 방어 시뮬(P6) 파고들기 깊이 |
| PR 강도 | Lv1 사실 · Lv2 · Lv3 최대 방어 | Lv2 | P5 문장 표현 수위. Lv3도 **날조 금지**, 오해 소지엔 ⚠️ 플래그 |
| 전문용어 강도 | Lv1 일반 · Lv2 · Lv3 전문 | 문서별 자동 | 포트폴리오=각주 허용/제출용=평이 용어 우선 |

**입력은 표준 폼으로.** 사용자가 채워야 할 값은 항상 `templates/intake-form.html`(동일 양식)으로 제공해
작성 부담을 낮춘다. AI가 뱅크 기반으로 글을 쓸 때는 **AI 티 없이, 전문 컨설턴트 문체**로(=`writing-voice.md`).

## P0 — 진입 결정 (Intake)

시작 시 3가지를 묻고 분기한다.

1. **타깃 JD 있음?** — (a) 있음: 붙여넣기/링크 → 5-D Prism 분해(methodology.md §4). (b) 없음, 찾아야 함: 직무·기업 키워드로 JD Browsing(로그인/페이월은 "인증 필요"로 정지). (c) 없음, 필요 없음 → 마스터본.
2. **맞춤형 vs 마스터본?** — 맞춤형=특정 JD에 정렬된 제출본. 마스터본=모든 경험을 정리한 durable 원본.
3. **기존 뱅크에서 파생?** — `.private/experience-bank.md` 존재 시 "재입력 없이 이 뱅크에서 뽑을까요?"(P8). 없으면 신규 구축(P1로).

## P1 — 타임라인 입력 (기간)

학력·병역·경력을 **년/월/일** 단위로 받는다(정확한 기간이 타임라인·연차 계산·공백 진단의 기준).

- 학력: 초/중/고/대학/석사/박사 — 입학·졸업 연월(일 단위 선택).
- 병역: 군별·계급·복무 분야·전역 유형·기간.
- 경력: 회사·직무·재직 기간(현재 진행 포함)·지역.
- 공백기(gap)는 자동 감지해 표시하되 **가치판단 없이** 사실만(사유는 사용자 입력 선택).

입력은 `intake-form.html`의 "타임라인" 섹션으로 수집.

## P2 — 경험 분류 (필살기 / 빌살기 / 밉살기)

methodology.md의 3분류를 뱅크에 적용. **모든 경험을 나열**하되 각 경험을 평가해 **해시태그**로 분류한다.

**필살기** = 유사경험 ∩ 성공경험 ∩ 인사이트(3조건 교집합). 개수 상한은 **연차별 차등**:

| 연차 | 필살기 권장 상한 | 근거 |
|---|---|---|
| 신입 (0년) | 1–2 | 경험 총량이 적음. 1개라도 깊게 |
| 주니어 (1–3년) | 2–3 | 대표 성공경험 중심 |
| 미들 (4–8년) | 3–4 | 도메인 대표작 + 리더십 |
| 시니어 (9년+) | 4–5 | 영역별 대표작. 과다 나열은 오히려 희석 |

> 상한은 권장치. 넘으면 "핵심 N개로 압축" 제안(약한 것은 빌살기로 강등).

**해시태그 분류 축(각 경험에 자동 부여):**
- `#레벨` — `#필살기` / `#빌살기`(무난·범용) / `#밉살기`(실패·약점, 배움으로 재구성)
- `#카테고리` — 직무 도메인(예: `#원가절감` `#공정개선` `#품질` `#리더십` `#데이터`)
- `#KPI버킷` — P3의 5버킷 중 해당(효율/품질/규모/재무/직무특화)
- `#Tier` — Truth Tier(T1 문서화 / T2 프레이밍 필요 / T3 미검증)
- `#도메인` — 산업/제품(예: `#자동차` `#모바일` `#반도체`) · `#스킬` — 기술(예: `#CATIA` `#PLC` `#Python`) · `#소프트스킬` — (예: `#리더십` `#협업` `#커뮤니케이션`)
- `#연차레벨` — 이 경험이 증명하는 레벨(신입/주니어/미드/시니어) · `#최신성` — 연도(최근일수록 가중) · `#증빙` — 유/무(URL·문서)

**태그 쿼리 파생(활용):** 태그로 **JD 맞춤 부분집합을 자동 조립**한다. 예) 원가 중심 JD → `#필살기 ∩ #재무 ∩ #T1`; 리더십 강조 → `#소프트스킬:리더십 ∩ #필살기`; 특정 도메인 지원 → `#도메인:반도체`. 파생(P8)·매칭(①)·문서(②)가 이 쿼리로 원자를 뽑아 재입력 없이 정렬한다.

**밉살기**도 동일하게 분류하되, 대외 문서엔 **배운점·개선 서사**로만 등장(약점 자체 나열 금지).

**정량화 부족 시** → 사용자와 대화로 채운다("이 성과, 수치로 말하면? 대략이라도 범위는?"). 못 채우면 T3로 두고 대외 미사용.

**경험별 평가표**(작업용): 각 경험을 아래 표로 보여주고 **무엇을 보완하면 필살기 급이 되는지** 처방.

| 축 | 점수(1–4) | 보완 포인트 |
|---|---|---|
| 유사성(타깃 직무 연관) | · | "JD의 X 요건과 연결하면 ↑" |
| 성공 근거(정량) | · | "수치·출처 확보 시 ↑" |
| 인사이트(배움·전이) | · | "그래서 무엇을 다르게 하는가" |
| 차별성 | · | "남과 다른 지점 1문장" |

마스터본 HTML 출력 시 **평가표 포함/미포함**을 사용자가 선택(제출용은 미포함).

## P3 — 포트폴리오화 (C·R·A·R·I + KPI + NCS)

각 경험을 5요소 골격으로 자동 구조화(methodology.md 5단계 확장).

**C·R·A·R·I:**
- **C**ontext — 상황·과제(왜 필요했나).
- **R**ole — 역할 + **기여도 %**(팀 성과를 개인 단독처럼 쓰지 않기 위한 장치).
- **A**ction — 실제 한 행동(액션 동사, 방법 명시).
- **R**esult — 결과(정량 우선, X-Y-Z 공식: "Y를 이뤄서 Z로 측정, X를 함으로써").
- **I**nsight — 배운 것·전이 가능한 통찰.

**KPI 2층 구조** — 재무만이 아니라 경험별·직무별 KPI를 확인·표준화:

1. **고정 5버킷**(항상 존재): `효율`(시간·속도·처리량) · `품질`(불량·정확도·재현) · `규모`(범위·인원·트래픽) · `재무`(비용·매출·ROI) · `직무특화`(도메인 고유).
2. **런타임 KPI명**(직무별 실명): JD Browsing으로 해당 직무의 표준 지표를 찾아 5버킷에 매핑(예: 제조=UPH·CapEx·OEE / 개발=p99·MTTR·가용성 / 마케팅=CAC·LTV·전환율).

**NCS 앵커**(국가직무능력표준): 한국 직무역량 표준을 참조해 KPI·역량 서술의 근거로 활용. **무조건 따를 필요 없음** — 리즈너블하거나 상대가 인정할 형태면 자유. runtime JD Browsing.

**증빙 링크** — 작업용에만 첨부(포트폴리오 URL·레포·수상 근거). **제출용은 자동 제외**. 두 모드를 구분 출력.

## P4 — 문체 게이트 & 낭독 가독성 루프

상세 규칙은 `writing-voice.md`. 요지:

- **문체 게이트(자동 적용)**: 생성 문장은 AI-tell 블랙리스트를 통과해야 함. 걸리면 재작성.
- **진지 문서 규칙**: 이력서·자소서 등은 이모지·과한 감탄·수사 금지. 명확·간결·읽기 쉬운 컨설턴트 문체.
- **낭독 가독성 루프**(선택): 사용자에게 완성 문단을 **소리 내어 2~3회 읽게** → STT로 받아 원문과 diff → 자주 걸리거나 바꿔 읽는 지점 = 어색한 구간 → 그 문장을 우선 개선. (STT 입력이 없으면 스킵.)

## P5 — 헤드라인 & 수치 무결성

- **헤드라인**: 문서 최상단 한 줄 요약(직무 정체성 + 대표 성과). 교체 가능한 후보 2–3안 제시, 차등(강/중/약) 톤.
- **수치 무결성(hard rule):**
  - 출처 라벨을 함께 관리(어느 프로젝트·기간의 수치인가).
  - **임의 합산 금지** — 서로 다른 프로젝트/카테고리 수치를 몰래 더해 부풀리지 않는다.
  - 합칠 땐 **카테고리/프로젝트 구분이 표면에 드러나게** 표현.
  - 근거 없는 수치는 만들지 않는다(T3는 대외 미사용).

## P6 — 방어 시뮬레이션 (모의 압박)

자동 게이트로 실행하되 **사용자가 스킵 가능**. evaluation.md 2단 구조를 그대로 사용.

- **두 단계 모두**: Stage 1 냉정 진단(실경쟁력 채점) → Stage 2 최대 방어 PR(진실 유지).
- **질문 경중**: 사용자가 상/중/하 선택(꼬리질문 파고들기 깊이).
- **면접관 페르소나**: 해당 분야 **전문가 면접관** 시점으로 꼬리질문(JD 있으면 JD 우선 정렬). 페르소나는 *질문 각도*를 위한 것이지 캐릭터 연기가 아니다.
- **PR 강도 Lv1–3**: 표현 수위. Lv3=가장 과감하되 방어 가능·날조 금지·오해 소지 ⚠️ 플래그.
- **전문용어 강도 Lv1–3**: 포트폴리오=각주 달고 전문어 허용 / 제출용=평이 용어 우선.
- **과정별 점수화**: evaluation.md 루브릭으로 매 답변 채점(자기소개 100 / 실무 16 / 임원·HR 100). 미달 항목엔 "왜 미달 + 다음 목표 1개".

## P8 — 지속 · 재사용 · 갱신 (Loop Close)

**저장 (SSOT)** — `.private/experience-bank.md`, **하이브리드 형식**:
```
## [경험 ID] 제목
<!-- meta: {연도:2023, 회사:"핀테크 A", 카테고리:[결제,인프라], KPI버킷:[효율,규모],
     레벨:필살기, Tier:T1, 기여도:70, 증빙:"url"} -->
- Context: …
- Role(70%): …
- Action: …
- Result: … [출처: …]
- Insight: …
```
사람이 읽는 마크다운 본문 + 경험별 구조화 메타 헤더(스킬이 재파싱). gitignore, 커밋 금지.

**재사용 (파생)** — 새 JD 도착 시 P0에서 "이 뱅크에서 파생?" → 전체 재입력 없이 부분집합 선택 + JD 정렬(순서·헤드라인·강조 재배치). 마스터=불변 소스, 맞춤형=그 위의 뷰.

**갱신 (자동 트리거)** — 세션마다 "새 경험/수치 생겼나요?" 자동 확인 후 증분 갱신:
- (a) 새 경험 추가 · (b) 정량화 보완 대화로 빈 수치 채움 · (c) **Truth Tier 승격**([T3]→증빙 확보 시 [T1], 스킬이 능동 **리마인드**) · (d) 연차/성과 반영 필살기 재평가.
- 변경 이력(무엇이 언제) 로그.

**지원 히스토리 (자소설닷컴 스타일)** — `.private/applications/`에 지원 건별 누적:
```
<!-- application: {회사, 직무, 지원일, 마감일, 전형단계, 결과, 파생본경로} -->
```
- 전형 단계 파이프라인: `관심 → 서류접수 → 서류합격 → 인적성/과제 → 1차면접 → 2차면접 → 최종합격/불합격`.
- 결과 뱃지: 진행중 / 합격 / 불합격 / 대기.
- **제출 내용물 보관**: 그 지원에 낸 파생본(이력서·자소서 문항+답변)을 스냅샷으로 저장 → 나중에 조회.
- **히스토리 누적**(덮어쓰지 않음). "지원 현황 보여줘" → `templates/application-tracker.html` 대시보드로 렌더(D-day·단계·결과 한눈에).

**루프 연결** — 뱅크는 **①–⑦ 전 태스크의 공통 근거 소스**(③면접=필살기 소비·복기 write-back, ⑥연봉=성과 근거 소비 포함). 각 태스크가 뱅크를 읽어 재활용(매번 처음부터 묻지 않음). 로드·파생·write-back·체이닝 규율은 SKILL §6.0 공통 계약.

## 실행 순서 요약 (제로베이스 → 취업까지)

```
P0 진입결정 → P1 타임라인 → P2 경험분류(필/빌/밉 + 해시태그 + 평가표)
   → P3 포트폴리오화(C·R·A·R·I + KPI 5버킷 + NCS) → P4 문체게이트/낭독
   → P5 헤드라인/수치무결성 → P6 방어시뮬(2단 채점) → P7 출력(작업/제출·HTML/A4)
   → P8 뱅크 저장·갱신 → [새 JD] 파생 반복 + 지원 히스토리 누적
```
첫 사용자는 P0–P8을 한 번 완주해 뱅크를 만들고, 이후 지원마다 P0→(파생)→P7만 반복한다.
